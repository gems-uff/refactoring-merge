# Project: pucomex-duimp-10978-src — Merge commit SHA1: e444c5baf82d565b7fe70c1ea296bc7b41016206

This merge commit contains **two** independent, well-supported cases in which the
merge-effort lines (`++`) were produced to resolve a conflict caused by a refactoring
that had already been introduced in one of the parents. Because the mandatory output
file name is keyed to the merge-commit SHA1, both cases are documented in this single
file, each under its own complete section.

# CASE 1 — Rename_Method

## Modified file(s):
- `pucomex-duimp-core/src/main/java/br/gov/serpro/pucomex/duimp/persistence/DuimpDAO.java` *(location of the rename)*
- `pucomex-duimp-core/src/main/java/br/gov/serpro/pucomex/duimp/business/integracao/carga/CargaBC.java` *(location of the merge effort)*

## Class(es) affected by the merge effort:
- `DuimpDAO`
- `CargaBC`

## Merge effort lines in the combined diff

Rename evidence in the parents (`DuimpDAO.java`) — note these are parent-context lines,
**not** merge effort:

```diff
 -	public DuimpVersao loadVerificacaoPermissaoAcesso(Duimp duimp) {
 +	public DuimpVersao loadVerificacaoPermissaoAcessoInterno(Duimp duimp) {
```

Merge effort at the conflicting call site (`CargaBC.recuperarDadosCE`):

```diff
+ 		Duimp duimp = new DuimpVersao();
+ 		duimp.setNumero(numero);
 -		duimp =  getDuimpDAO().loadVerificacaoPermissaoAcesso(duimp);
++		duimp =  getDuimpDAO().loadVerificacaoPermissaoAcessoInterno(duimp);
```

Reading the notation:
- ` -` (space-minus) → line present in **Parent 2**, discarded in the merge.
- ` +` (space-plus) → line present in the merge result and in **Parent 1** (col-1 unchanged).
- `+ ` (plus-space) → line added relative to **Parent 1**; present in **Parent 2** and the result.
- `++` → line **created in the merge** (merge effort).

## Number of merge-effort lines (++ and --) associated with the refactoring types under analysis:
**1 line** (`++` — the adapted call `loadVerificacaoPermissaoAcessoInterno(duimp)`).

## What each side had
- **Parent 1** (evidence of the refactoring): in `DuimpDAO.java` the DAO method was already
  named `loadVerificacaoPermissaoAcessoInterno` (` +` line, unchanged w.r.t. P1). Parent 1 did
  **not** contain the `Duimp duimp = new DuimpVersao()` block in `CargaBC.recuperarDadosCE`
  (those lines are `+ `, i.e. added relative to P1).
- **Parent 2**: in `DuimpDAO.java` the method still used the old name
  `loadVerificacaoPermissaoAcesso` (` -` line, discarded). In `CargaBC.recuperarDadosCE`,
  Parent 2 had **added a new caller** that invoked the method under its old name —
  `getDuimpDAO().loadVerificacaoPermissaoAcesso(duimp)` (` -` line, discarded).

The rename `loadVerificacaoPermissaoAcesso` → `loadVerificacaoPermissaoAcessoInterno` was
therefore introduced in **Parent 1**.

## Interpretation
This is a **Rename_Method** refactoring (`loadVerificacaoPermissaoAcesso` →
`loadVerificacaoPermissaoAcessoInterno`) introduced in Parent 1, evidenced directly by the
parent-side `DuimpDAO.java` lines (` -` old name in P2, ` +` new name in P1/result).

Independently, Parent 2 added a new call site in `CargaBC.recuperarDadosCE` that still used
the **old** method name. When the two branches were merged, Parent 2's freshly added caller
referenced a method name that no longer existed after Parent 1's rename. The developer
resolved this by rewriting the call with the renamed identifier, producing the single `++`
merge-effort line `duimp = getDuimpDAO().loadVerificacaoPermissaoAcessoInterno(duimp);`.

The case is well-supported because:
- the `++` merge-effort line **literally contains the renamed method**, and the discarded
  ` -` line shows the pre-rename name on Parent 2's side, so the merge-effort line is
  unambiguously the resolution of the rename conflict (not inferred from final code only);
- the refactoring is evidenced by **parent-side differences** in a *different* file
  (`DuimpDAO.java`), confirming it was performed in a parent and not by the merge itself;
- the merge itself only adapted the caller — it did not perform the rename.

## Relevant final code in the merge

```java
// ImplCancelamentoBC.verificacaoCargaAtracadaDuimpRegistrada
DadosCarga dadosSistemaCarga = cargaBC.obterDadosCarga(
        DadosCargaRequest.builder()
        .fromDuimp(duimp)
        .gerandoExceptionQualquerErro() // equivale ao false na chamada antiga (ce, booleano)
        .build());
duimp.getCarga().setDadosCargaCCT(dadosSistemaCarga);
if (dadosSistemaCarga == null) {
    DuimpMessages.geraException(DuimpMessages.INTEGRACAO_ERRO_OBTER_CARGA);
} else if (dadosSistemaCarga.getDataAtracacaoEscala() != null) {
    DuimpMessages.geraException(DuimpMessages.DUIMP_CARGA_JA_ATRACADA);
}
```

## Number of merge-effort lines (++ and --) associated with the refactoring types under analysis:
**5 lines** for the adapted call (the `DadosCarga dadosSistemaCarga = cargaBC.obterDadosCarga( … .build());`
statement), **+1 line** for the supporting `++import … DadosCargaRequest;`. **Total: 6 `++` lines.**

## What each side had
- **Parent 1**: in `ImplCancelamentoBC` Parent 1 had *added* the new method
  `verificacaoCargaAtracadaDuimpRegistrada(...)` (the method body lines are ` +`, i.e. present
  in P1 and the result), and inside it it called the carga service with the **old two-argument
  signature**: `cargaBC.obterDadosCarga(duimp.getCarga().getIdentificacaoCarga(), true)`
  (`- ` line, discarded). Parent 1 also still used the old two-argument form in
  `CargaBC.recuperarDadosCE` (`- carga.setDadosCargaCCT(obterDadosCarga(... , true));`).
- **Parent 2** (evidence of the refactoring): in `CargaBC.recuperarDadosCE`, Parent 2 had
  already replaced the old two-argument call with the new **single-parameter** form,
  `obterDadosCarga(dadosCargaRequest)`, building a `DadosCargaRequest` via
  `DadosCargaRequest.builder().fromDuimp(...).gerando…().build()` (`+ ` lines).

The signature change from `obterDadosCarga(String ce, boolean geraException)` to
`obterDadosCarga(DadosCargaRequest)` was therefore introduced in **Parent 2**.

## Interpretation
The evidenced refactoring is a **Merge_Parameter** (the classic *Introduce Parameter Object*
form, which collapses several parameters into one): the two existing arguments of
`CargaBC.obterDadosCarga(...)` — the carga identifier (`String ce`) and the error-handling
flag (`boolean`) — are fused into a single semantically unified parameter of type
`DadosCargaRequest`. The two original parameters are not merely dropped; they are both
preserved inside the new object (`.fromDuimp(...)` carries the identifier, and
`.gerandoExceptionQualquerErro()` / `.gerandoExceptionApenasErrosDesconhecidos()` carries the
boolean behaviour). The diff even documents the fusion explicitly with the in-line comment
`// equivale ao false na chamada antiga (ce, booleano)` (“equivalent to the `false` in the old
`(ce, boolean)` call”), which makes the two-into-one mapping self-evident.

Why this is attributed to a parent and not to the merge: the new single-parameter signature is
present on **Parent 2's** side (the `+ ` lines in `CargaBC.recuperarDadosCE` already call
`obterDadosCarga(dadosCargaRequest)`), while the old two-parameter call form is on **Parent 1's**
side (the `- ` lines). Parent 1 independently added a *new* caller
(`verificacaoCargaAtracadaDuimpRegistrada`) that still used the old two-parameter signature.
At merge time, Parent 1's new caller no longer matched Parent 2's refactored signature, so the
developer rewrote the call into the `DadosCargaRequest.builder()…build()` form — the `++`
merge-effort lines — and added the corresponding `++import … DadosCargaRequest;`.

The case is well-supported because the `++` merge-effort lines carry the refactored
single-parameter signature (with the explanatory comment tying them to the old two-argument
call), the pre-refactoring two-argument form is visible on the parent-context (`- `) lines, and
the new signature is independently evidenced on Parent 2 in `CargaBC`. (A stricter reviewer may
prefer to label this **Change_Parameter_Type**; the two-into-one fusion documented by the
comment is the reason `Merge_Parameter` is reported as the primary type.)
