## What each side had

**Parent 1 (P1)** already introduced `NameNodeRpcServer` as a separate class that owns `rpcAddress` and `serviceRPCAddress` fields, exposing them via `getRpcAddress()` and `getServiceRpcAddress()` methods. P1's `NameNode` no longer had these fields directly and accessed them through `rpcServer`.

**Parent 2 (P2)** still had `rpcAddress` and `serviceRPCAddress` as `protected` fields on `NameNode` itself (`protected InetSocketAddress rpcAddress = null; protected InetSocketAddress serviceRPCAddress = null;`), and the log statements after `activate(conf)` used the fields directly: `LOG.info(... + rpcAddress)` and `LOG.info(... + serviceRPCAddress)`.

In `BackupNode`, P2 used `nnReg.getAddress() + " expecting " + rpcAddress` while P1 had refactored to use `nnRpcAddress` as a local field on the inner `BackupNodeRpcServer`.

## Interpretation

This case primarily exhibits **Move_Attribute**: `rpcAddress` and `serviceRPCAddress` were moved from the `NameNode` class to `NameNodeRpcServer` in P1 before the merge. P2 still referenced them as `NameNode` fields. The merge had to reconcile the two approaches:

- The `--` lines show P2's two field declarations (`protected InetSocketAddress rpcAddress = null; protected InetSocketAddress serviceRPCAddress = null;`) being removed from `NameNode`.
- The `++` lines show the merged log statements using `rpcServer.getRpcAddress()` and `rpcServer.getServiceRpcAddress()` — the delegating accessor pattern introduced by P1's move.

The `BackupNode` case further confirms this: the reference to `rpcAddress` in the journal exception message (`--  + rpcAddress`) had to be updated to `++ nnRpcAddress` (the field now owned by `BackupNodeRpcServer`).

Additionally, P1 introduced `nn.checkOperation(OperationCategory.*)` calls throughout `NameNodeRpcServer`, which P2 did not have. These appear as `++` lines in numerous RPC methods, representing merge effort to add the new operation-category guards consistently.
