## Relevant final code in the merge

```java
// LocalStore.java
public abstract static class BinaryAttachmentBody implements Body {
    protected String mEncoding;

    public abstract InputStream getInputStream() throws MessagingException;

    @Override
    public void writeTo(OutputStream out) throws IOException, MessagingException {
        InputStream in = getInputStream();
        try {
            boolean closeStream = false;
            if (MimeUtil.isBase64Encoding(mEncoding)) {
                out = new Base64OutputStream(out);
                closeStream = true;
            } else if (MimeUtil.isQuotedPrintableEncoded(mEncoding)) {
                out = new QuotedPrintableOutputStream(out, false);
                closeStream = true;
            }
            try {
                IOUtils.copy(in, out);
            } finally {
                if (closeStream) { out.close(); }
            }
        } finally {
            in.close();
        }
    }

    @Override
    public void setEncoding(String encoding) throws MessagingException {
        mEncoding = encoding;
    }

    public String getEncoding() { return mEncoding; }
}
```

**~30 lines** (++ for the new `BinaryAttachmentBody` abstract class declaration, its `writeTo`, `setEncoding`, `getEncoding` methods, the updated `TempFileBody extends BinaryAttachmentBody` and `LocalAttachmentBody extends BinaryAttachmentBody` declarations removing the now-inherited `mEncoding` field and `setEncoding`; -- for P2's standalone `TempFileBody implements Body` and P1's `LocalAttachmentBody implements Body` with duplicated `mEncoding` field)

## What each side had

**Parent 1 (P1)** had `LocalAttachmentBody` declared as `implements Body` directly, with an instance field `protected String mEncoding` and a `setEncoding()` method defined on the class itself. `TempFileBody` did not exist in P1; instead P1 used `LocalAttachmentMessageBody` (which extended `LocalAttachmentBody`) for message bodies, and attachment body parts were created inline as `new LocalStore.TempFileBody(attachment.filename)` (which apparently existed in the base but differently). The `writeTo()` method in `LocalAttachmentBody` encoded directly using `Base64OutputStream`.

**Parent 2 (P2)** had `TempFileBody` declared as a separate standalone inner class implementing `Body` directly, with its own `writeTo()` logic (also using `Base64OutputStream` directly). `LocalAttachmentBody` was a separate class also directly implementing `Body`, also with its own `mEncoding` field. Both classes duplicated the encoding/writeTo logic.

## Interpretation

This is a clear **Extract_Class** refactoring. P2 had introduced `TempFileBody` as a sibling to `LocalAttachmentBody`, and both classes duplicated the binary encoding logic. The merge extracted the shared functionality into a new abstract superclass `BinaryAttachmentBody`, which now holds the `mEncoding` field, `setEncoding()`, `getEncoding()`, and a generalized `writeTo()` method. Both `TempFileBody` and `LocalAttachmentBody` are refactored to `extend BinaryAttachmentBody`, eliminating the duplication.

The `--` lines confirm P2's standalone `TempFileBody implements Body` with its own `writeTo()` using `Base64OutputStream` directly. The `++` lines show `BinaryAttachmentBody` as the new abstraction, and both `TempFileBody` and `LocalAttachmentBody` now `extend BinaryAttachmentBody`. The field `protected String mEncoding` was lifted from `LocalAttachmentBody` to `BinaryAttachmentBody` (a `Pull_Up_Attribute`). The call site in `MessageCompose` is updated to use `TempFileBody` and `TempFileMessageBody` instead of `LocalAttachmentBody`/`LocalAttachmentMessageBody`, using the wider `Body` type for the local variable.
