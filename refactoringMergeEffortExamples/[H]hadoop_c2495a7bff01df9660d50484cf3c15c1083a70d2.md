## Number of merge-effort lines (++ and --) associated with the refactoring types under analysis:
**~30 lines** (++ and -- directly tied to the `updateHeartbeat` / `handleHeartbeat` signature changes and call-site reconciliation)

## What each side had

**Parent 1 (P1)** had `DatanodeDescriptor.updateHeartbeat` accepting `StorageReport[] reports, int xceiverCount, int failedVolumes` — it had already introduced `StorageReport[]` but without cache parameters. `DatanodeManager.handleHeartbeat` likewise had `StorageReport[] reports` but without `cacheCapacity`/`cacheUsed`. The DatanodeDescriptor constructor passed `StorageReport.EMPTY_ARRAY, xceiverCount, failedVolumes`.

**Parent 2 (P2)** had the old pre-`StorageReport` API: `updateHeartbeat(long capacity, long dfsUsed, long remaining, long blockPoolUsed, long cacheCapacity, long cacheUsed, int xceiverCount, int failedVolumes)` — a signature with 8 primitive parameters. The `handleHeartbeat` accepted individual `long` disk metrics instead of `StorageReport[]`.

The two branches evolved the same method signature in incompatible directions: P1 adopted `StorageReport[]` (collapsing capacity/dfsUsed/remaining/blockPoolUsed into a report array) but dropped cache params, while P2 had the old individual longs including `cacheCapacity`/`cacheUsed`. The merge had to produce a unified signature that accepts `StorageReport[]` (from P1) **and** `cacheCapacity`/`cacheUsed` (from P2), resulting in the `++` lines.

## Interpretation

This is a clear **Change_Parameter_Type** case. The `--` lines demonstrate two incompatible parameter lists for the same methods (`updateHeartbeat`, `handleHeartbeat`, `HeartbeatManager.updateHeartbeat`):
- P1's form: `(StorageReport[] reports, int xceiverCount, int failedVolumes)` — already aggregated storage
- P2's form: `(long capacity, long dfsUsed, long remaining, long blockPoolUsed, long cacheCapacity, long cacheUsed, int xceiverCount, int failedVolumes)` — old individual primitives

The merge created the `++` unified form: `(StorageReport[] reports, long cacheCapacity, long cacheUsed, int xceiverCount, int volFailures)`, combining P1's `StorageReport[]` with P2's explicit `cacheCapacity`/`cacheUsed` parameters. This is confirmed by consistent `++` call-site updates across `HeartbeatManager`, `DatanodeManager`, `FSNamesystem`, and `NameNodeRpcServer`.
